什么是Operator

倪克松 2023-04-25 12:27:10
Categories： Kubernetes学习笔记 Tags：

1. 为什么需要Operator

随着K8s的广泛应用，传统的K8s资源、组件等已经无法满足自定义的需求。K8s提供了接口允许用户自己定义K8s中的资源和控制器来增强K8s的功能，让集群更加的智能化，让运维人员可以更省心。所以Operator模式的本质就是”让K8s学会自己运维自己“，他把一个领域运维人员的运维经验，也就是把他们所维护的应用应该怎么部署、出现异常了应该怎么恢复这一系列运维过程在Kubernetes上的操作程序化，并交给自定义控制器去实施，让用户能够通过自定义资源来管理自己的应用。

要理解它的实现机制，就需要先理解 CR、CRD、Controller（其他的后面再理解）。

此处做最简单易懂的描述：

  • CR就是customer resource，用户自定义的K8s资源。什么是K8s资源？就是类似Pod，Deplyment，Replica Set等，我们可以定义一个类似于Deployment的东西（众所周知这是一个Pod Controller），来管理一系列的Pod。所以你可以在心里直接理解为，CR就是我自己写的一个定制化的Deployment。
  • CRD就是CR的defination，定义CR的东西。
  • Controller就是类似K8s原生的那个Controller-Manager的东西，我们可以写一个自定义的Controller-Manager做Pod状态的监控，调整控制等。

官方解释很清晰

CR

所谓自定义资源，就是用户可以提交一个类似Deployment声明的结构定义给Kubern-etes，比如上文的SampleDB，所以我们也可以通过kubectl get SampleDB之类的命令来查询SampleDB。而SampleDB中可以定义各种控制具体的数据库应用所需的字段，类似Deployment中的各种属性，比如常见的replicas控制副本数等。

Controller能够做的事情就类似于kube-con-troller-manager ，可以获取自定义资源中的声明内容，通过调用apiserver来管理集群的实际状态，比如查询ConfigMap内容来获取数据库连接地址，创建一个State-fulSet来启动数据库实例等。

Controller

一个控制器至少追踪一种类型的 Kubernetes 资源。这些对象有一个代表期望状态的 spec 字段。 该资源的控制器负责确保其当前状态接近期望状态。Kubernetes 内置一组控制器，运行在kube-controller-manager内， 这些内置的控制器提供了重要的核心功能。

你会遇到某些控制器运行在控制面之外，用以扩展 Kubernetes。 或者，如果你愿意，你也可以自己编写新控制器。 你可以以一组 Pod 来运行你的控制器，或者运行在 Kubernetes 之外。 最合适的方案取决于控制器所要执行的功能是什么。

就是，原始的资源被controller manager控制，但是这时候你想自己写一些资源（CR），那么这些自定义资源也需要一个类似controller manager的东西管理啊！那就是自己写一个controller！

我的理解，他们三个是这样的：

首先用户操作kubectl输入命令对资源X做操作，api-server反映资源X的状态，当自定义的controller监听到相应的任务时会使用其内部的CR（Pod控制器）操作相应的Pod，把操作后的结果写入api-server，此时Node中的Kubelet看到了它应该做的工作，于是去操作相应的Pod。

同时，当集群运行的过程中Pod的状态不符合CD的规定/预期的时候，同样会进行上述的过程，完成对Pod状态的约束控制。

2. 控制器模式

控制器模式在日常生活中的应用非常广泛.我们先来思考一下夏天空调制冷时的工作流程：

  1. 启动后设置一个温度，比如25℃。
  2. 空调检测室温，确定室温是否高于25°℃。
  3. 如果高于25℃，就开始制冷，过一会儿再监测室温变化。
  4. 如果不高于25C，就保持静默，直到室温超过25C。

这其实就是控制器模式的典型工作流程，外部输入一个“期望值”，一个“控制器”不断观测“环境状态”的“实际值”和“期望值”之间的差异，然后不断调整“实际值”向“期望值”靠拢。Kubernetes中通过“声明式API”定义了一系列的“资源对象（CR）”，然后通过很多“控制器（Controller）”来“调整”这些资源对象的实际状态向期望状态靠拢，从而实现整个集群“尽可能”靠拢配置中声明的期望状态。

控制回路

在机器人技术和自动化领域，控制回路（Control Loop）是一个非终止回路，用于调节系统状态。

在任务执行时，集群随时都可能被修改，并且控制回路会自动修复故障。 这意味着很可能集群永远不会达到稳定状态。只要集群中的控制器在运行并且进行有效的修改，整体状态的稳定与否是无关紧要的。例如，如果你使用一个控制回路来保证集群中有足够的Pod，那么控制器就需要当前集群外的 一些服务在需要时创建新Pod。Pod控制器会让Pod的当前状态不断接近期望状态。

3. CR和CRD

定义：

  • CRD：Custom Resource Definition，允许用户自定义kubernetes资源（CR）。
  • CR：Custom Resource，CRD是一个类型，CR是它的的具体实例。

创建CR的方式有两种，一种是通过CRD，一种是通过聚合API，聚合API这里仅作了解：

CRD 聚合 API
无需编程。用户可选择任何语言来实现 CRD 控制器。 需要编程，并构建可执行文件和镜像。
无需额外运行服务；CRD 由 API 服务器处理。 需要额外创建服务，且该服务可能失效。
一旦 CRD 被创建，不需要持续提供支持。Kubernetes 主控节点升级过程中自动会带入缺陷修复。 可能需要周期性地从上游提取缺陷修复并更新聚合 API 服务器。
无需处理 API 的多个版本；例如，当你控制资源的客户端时，你可以更新它使之与 API 同步。 你需要处理 API 的多个版本；例如，在开发打算与很多人共享的扩展时。
  • 资源（Resource） 是Kubernetes API中的一个端点， 其中存储的是某个类别的API对象的一个集合。 例如内置的 Pod 资源包含一组 Pod 对象。

什么是资源！我们kubectl get pods，然后获得了很多pod信息，那么下面具体的多个pod就是Pod对象，这里的“Pod”这个名字，就是一种“资源”。同理我们可以kubectl get deployment，那么deployment这种pod controller也是一种资源，而具体的我们创建的一个个的deployment就是这种资源的对象。

  • 定制资源CR（Custom Resource） 是对 Kubernetes API 的扩展，不一定在默认的 Kubernetes 安装中就可用。定制资源所代表的是对特定 Kubernetes 安装的一种定制。 不过，很多 Kubernetes 核心功能现在都用定制资源来实现，这使得 Kubernetes 更加模块化。

4. Operator的组成

webhook是真正完成业务逻辑的两个组件

  • webhook：webhook关联在apiserver上，是一种http回调。在api-server特定事件发生时查询已注册的webhook，把相应的消息转发到某url。按照处理类型的不同，用户一般可以定义两类webhook，分别是mutating webhook(变更传入对象)和validating webhook(传入对象校验)

  • controller：controller的核心是工作队列。controller是监测集群状态变化，并据此作出相应处理的控制循环。它关联一个工作队列，循环处理队列内容。每一个controller都试图把集群状态向预期状态推动，不同Controller关注的对象不同，如replicaset controller（关注副本数）和endpoint controller（关注service和对应的pod副本的变化）。

    • 工作队列: controller核心组件，controller 会监控集群内关注的资源对象的变化，并把自己关心的对象的事件(动作和key)，存储于工作队列中。

  • operator：operator是描述、部署和管理kubernetes应用的一套机制，从实现上来说，operator =CRD + webhook + controller

Operator framework给用户提供了webhook和controller框架，包括消息通知、失败重新入队等等，开发人员仅需关心被管理应用的运维逻辑实现。

主流Operator framework项目

  • kubebuilder: https:/lgithub.com/kubernetes-sigs/kubebuilder
  • operator-sdk: https:/lgithub.com/operator-framework/operator-sdk

两者没有本质上的区别，都是使用的官方的controller-tools 和controller-runtime。细节上kubebuilder相应的测试、部署、代码生成工具更完善，如Makefile和 Kustomize等工具的集成。operator sdk的ansible operator、Operator Lifecycle Manager等上层支持更好些
