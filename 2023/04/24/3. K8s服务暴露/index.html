K8s服务暴露

倪克松 2023-04-24 18:17:15
Categories： Kubernetes学习笔记 Tags：

1. Service

服务暴露就是指把运行在K8s集群中的软件提供的服务（功能）拿出来，在前端发出请求时做出相应的响应。比如我们通常开发的bs应用前端的浏览器页面发出get请求到后端，后端接收到请求后就会计算分析，得到结果，并返回给前端一个html页面。然而在K8s中，应用程序被“层层包裹”，放在了集群中的Node中的Pod中的容器中，我们前端应该如何访问它呢？

无论前端是怎样的结构，请求都一定是通过网络传输传来的，所以为了搞清楚K8s中的软件的服务是如何提供的，我们就必须了解K8s中Node的网络结构。

我们之前提到过在每一个Pod中都有一个Pause容器，它随Pod的创建而创建，掌握了此Pod的网络栈和存储资源，同一个Pod里的容器之间仅需通过localhost就能互相通信。那我们能不能把这个网络资源暴露出去呢？

q: 我们能把Pod的地址转为公网ip暴露给外部吗？

经过思考我们就会发现是不可以的，因为Pod的生命是脆弱的，它随时有可能被销毁，重建。当Pod运行出现问题重启后地址会改变，但是服务的地址应该是稳定的。

为了解决这个问题，kubernetes提供了Service资源，Service会对提供同一个服务的多个pod进行聚合，并且提供一个统一的入口地址，是否属于同一个Service可以用Pod的标签进行区分，Service使用标签选择器实现聚合。如此，通过访问Service的入口地址就能访问到后面的pod服务。

Service其实是一个虚拟的概念，它并不是一个实际的组件，这个功能是由Node的Kube Proxy组件实现的。所以实际的框图如下所示。其中选中的Pod为与当前请求要求的标签一致的Pod。

创建Service分为以下几步：

  1. 用户通过Kubectl把消息写入api-server。
  2. api-server把这段消息写入etcd。
  3. api-service开始反映etcd中service的状态变化。
  4. KubeProxy读到api-server中的service信息后将信息转换为对应的访问规则。

2. Service类型和使用方法

Service的基本配置方式如下所示。类型有ClusterIP，NodePort，LoadBalancer，ExternalName应对不同场景。

kind: Service  # 资源类型
apiVersion: v1  # 资源版本
metadata: # 元数据
  name: service # 资源名称
  namespace: dev # 命名空间
spec: # 描述
  selector: # 标签选择器，用于确定当前service代理哪些pod
    app: nginx
  type: # Service类型，指定service的访问方式
  clusterIP:  # 虚拟服务的ip地址
  sessionAffinity: # session亲和性，支持ClientIP、None两个选项
  ports: # 端口信息
    - protocol: TCP 
      port: 3017  # service端口
      targetPort: 5003 # pod端口
      nodePort: 31122 # 主机端口

ClusterIP

默认值，它是Kubernetes系统自动分配的虚拟IP，只能在集群内部访问，使用ClusterIP时配置文件如上所示。。

在某些场景中，开发人员可能不想使用Service提供的负载均衡功能，而希望自己来控制负载均衡策略，针对这种情况，kubernetes提供了HeadLiness Service，这类Service不会分配Cluster IP，如果想要访问service，只能通过service的域名进行查询。使用这种模式时，配置文件如下所示。

apiVersion: v1
kind: Service
metadata:
  name: service-headliness
  namespace: dev
spec:
  selector:
    app: nginx-pod
  clusterIP: None # 将clusterIP设置为None，即可创建headliness Service
  type: ClusterIP
  ports:
  - port: 80    
    targetPort: 80

NodePort

将Service通过指定的Node上的端口暴露给外部，通过此方法，就可以在集群外部访问服务。NodePort的工作原理其实就是将service的端口映射到Node的一个端口上，然后就可以通过NodeIp:NodePort来访问service了。

配置文件如下：

apiVersion: v1
kind: Service
metadata:
  name: service-nodeport
  namespace: dev
spec:
  selector:
    app: nginx-pod
  type: NodePort # service类型
  ports:
  - port: 80
    nodePort: 30002 # 指定绑定的node的端口(默认的取值范围是：30000-32767), 如果不指定，会默认分配
    targetPort: 80

LoadBalancer

LoadBalancer和NodePort很相似，目的都是向外部暴露一个端口，区别在于LoadBalancer会在集群的外部再来做一个负载均衡设备，而这个设备需要外部环境支持的，外部服务发送到这个设备上的请求，会被设备负载之后转发到集群中。

ExternalName

ExternalName类型的Service用于引入集群外部的服务，它通过externalName属性指定外部一个服务的地址，然后在集群内部访问此service就可以访问到外部的服务了。
