curl方式和raw方式访问api server

倪克松 2023-04-27 21:47:16
Categories： Kubernetes学习笔记 Tags：

Kubernetes中的资源本质就是一个API对象，这个对象的“期望状态”被API Server保存在ETCD中，然后提供RESTful接口用于更新这些对象。我们可以直接和API Server交互，使用“声明”的方式来管理这些资源（API对象），也可以通过kubectl这种命令行工具，或者client-go这类SDK。

我们可以在Kubernetes的官网中找到API文档：https://kubernetes.io/docs/reference/generated/kubernetes-api/v1.23/。

因为各个组件都是通过与api-server交互做的功能实现，所以如何与api-server做交互是扩展开发中的重中之重。下面我们以一个Deployment资源的增删改查操作为例，看一下如何与API Server交互。

1. Curl方式访问API

我们通过kubectl proxy命令来暴露API Server服务，然后通过curl命令去创建和删除Deployment类型的资源实例。

由于kube-apiserver默认提供的是HTTPS服务，而且是双向TLS认证，而我们目前的关注重点是API本身，因此先通过kubectl来代理API Server服务：

[root@master ~]# kubectl proxy  --port=8080
Starting to serve on 127.0.0.1:8080

新开一个shell，可以通过简单的http请求来和API server交互了

[root@master ~]# curl localhost:8080/version
{
  "major": "1",
  "minor": "17",
  "gitVersion": "v1.17.4",
  "gitCommit": "8d8aa39598534325ad77120c120a22b3a990b5ea",
  "gitTreeState": "clean",
  "buildDate": "2020-03-12T20:55:23Z",
  "goVersion": "go1.13.8",
  "compiler": "gc",
  "platform": "linux/amd64"
}

我们还需要一个配置文件来描述Deployment资源，在本地创建一个nginx-deploy.yaml文件，内容如下：

piVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx

我们不再通过之前的方式apply这个yaml实现部署，通过api调用来操作这个Deployment。

执行下面的命令再default命名空间下创建一个Deployment

[root@master ~]# curl -X POST -H 'Content-Type: application/yaml' --data-binary '@nginx-deploy.yaml' http://localhost:8080/apis/apps/v1/namespaces/default/deployments

返回信息如下

{
  "kind": "Status",
  "apiVersion": "v1",
  "metadata": {
    
  },
  "status": "Failure",
  "message": "deployments.apps \"nginx-deploy\" already exists",
  "reason": "AlreadyExists",
  "details": {
    "name": "nginx-deploy",
    "group": "apps",
    "kind": "deployments"
  },
  "code": 409

此时能够get到这个deployment了：

}[root@master ~]#kubectl get deploymentNAME           READY   UP-TO-DATE   AVAILABLE   AGE
nginx-deploy   3/3     3            3           4s

和Deployment的创建类似，同样可以利用Kubernetes API来删除资源。Deployment的删除API是：

[root@master ~]# curl -X DELETE -H 'Content-Type: application/yaml' --data '
> gracePeriodSeconds: 0
> orphanDependdents: false
> '  http://localhost:8080/apis/apps/v1/namespaces/default/deployments/nginx-deploy

相应的Status类型的对象，我们可以通过这段信息清晰的看到被删除的对象所对应的grou、kind、name、uid等详细信息。

{
  "kind": "Status",
  "apiVersion": "v1",
  "metadata": {
    
  },
  "status": "Success",
  "details": {
    "name": "nginx-deploy",
    "group": "apps",
    "kind": "deployments",
    "uid": "b16c1e08-f1e4-45a7-913b-cc07323ec25b"
  }
}

2. Kubectl raw方式访问API

直接使用kubectl xxx–raw来访问API

请求

[root@master ~]# kubectl get --raw /version

返回信息

{
  "major": "1",
  "minor": "17",
  "gitVersion": "v1.17.4",
  "gitCommit": "8d8aa39598534325ad77120c120a22b3a990b5ea",
  "gitTreeState": "clean",
  "buildDate": "2020-03-12T20:55:23Z",
  "goVersion": "go1.13.8",
  "compiler": "gc",
  "platform": "linux/amd64"
}

如以上代码所示，通过kubectl get --raw可以实现和curl类似的效果，只是我们不需要指定API Server的地址了，同样认证信息也不需要，这时默认用了kubeconfig中的连接信息。

用这种方式查询nginx-deployment：

[root@master ~]# kubectl get --raw /apis/apps/v1/namespaces/default/deployments/nginx-deploy

返回的信息比较多

3. 总结分析

  • kubectl get–raw这种方式和kubectl proxy+curl本质是一样的，只是不需要再开两个窗口分别执行两条命令了。只是不需要指定API Server的地址了，同样认证信息也不需要，这时默认用了kubeconfig中的连接信息。

  • 交互中api返回的信息是json文件格式的对象信息。
